How I Built SplitSecond

The prompts behind a full expense-splitting web app — built from scratch, in a single conversation with Claude.

This is an overview, not a transcript. The actual conversation had well over 40 prompts — the cards below are paraphrased summaries of the key ones, written to be easy to follow for someone who wasn't there. The real messages were often shorter, messier, and more back-and-forth.

The app is real though — built in a single session with Claude, no prior code, deployed at split-second-weld.vercel.app. Firebase backend, works on mobile, fully shareable.

01 Design reference

Abhi said

I have a screenshot of a mobile UI I like the look and feel of. Can you generate a design.md file from it? I want to capture the colours, fonts, spacing, and overall visual language so I can use it as a reference going forward.

What happened

Claude read the screenshot and produced a full design spec — warm off-white background #F2EFE9, a single orange accent #E8471A, DM Serif Display for headings, DM Sans for body, and DM Mono for numbers. This file became the visual brief for everything that followed.

design.md
02 Concept + scope

Abhi said

I want to build an expense splitting app — something I can actually use with friends and family for dinners and trips. Not recurring bills, just fast in and out. Let's use that design.md we just made. What should this include? What should we call it?

What happened

Claude proposed a feature set and name options. We settled on QuickSplit as the working name (renamed later). The scope was locked: a home screen with recent activity, a new split creation flow, multiple expense types, a summary with debt simplification, and sharing. No mandatory accounts.

03 Architecture

Abhi said

I want this as a single HTML file — no build step, no frameworks. But it needs to sync in real time so two people on their phones can use it at the same time. Can we use Firebase? Also, anonymous login first — no forced account creation.

What happened

Architecture confirmed: Firebase Realtime Database for live sync, localStorage as a fallback. Auth would be anonymous-first — the primary CTA on the sign-in screen is "Use without an account" in orange. Creating an account is secondary. A greedy min-cash-flow algorithm was chosen for settling debts.

04 First build

Abhi said

I don't have a Firebase project set up yet — just use a placeholder config for now. Let's move. Build the whole thing.

What happened

Claude built the entire app in one pass — a single HTML file with a working home screen, split creation flow, expense entry, four split modes (equal, shares, exact, percentage), people management, a summary with debt graph, and QR-based sharing. Design tokens from design.md applied throughout.

UI Iteration
05 Visual refinement

Abhi said

The buttons don't feel right — "Add Person" and "Start Split" look equally important. Start Split should be the primary action, clearly orange. Add Person should be secondary — outlined, not filled.

What happened

"Start Split" became the full-orange primary. "Add Person" was restyled as a white, bordered secondary button. This button hierarchy pattern was then applied consistently across all flows throughout the app.

06 UI feedback

Abhi said

The hover states feel off — some elements have them, some don't. And when a modal opens on top of another modal, the stacking is broken. Also some text is misaligned in the expense rows.

What happened

Hover states were standardised across all interactive elements. The modal stacking issue was fixed with proper z-index layering. Expense row alignment was corrected to use consistent baseline alignment throughout.

07 Typography consistency

Abhi said

The amounts in the summary screen are using the wrong font — they should be in DM Mono like we specified in the design file. And the section headings inside modals feel too heavy.

What happened

All monetary amounts and debt figures were updated to DM Mono. Modal section headings were lightened to weight 500 to match the editorial hierarchy from the design spec.

New Features
08 Auth layer

Abhi said

We need actual login screens now — anonymous first, but also the option to sign in with email or Google so your splits follow you across devices. Add Firebase Auth to the app.

What happened

Firebase Auth was wired in with three paths: continue anonymously, sign in with Google, or create an email account. The anonymous session upgrades seamlessly when the user decides to create a real account — their existing splits carry over. The Firebase script block was fully replaced to expose auth globally.

09 Rename

Abhi said

I want to rename the app from QuickSplit to SplitSecond. Update everything — the title, the logo, the meta tags, all of it.

What happened

Every reference to QuickSplit across the HTML was updated to SplitSecond. The page title, app header, manifest name, share text, and all meta tags were updated in one pass.

10 Share as image

Abhi said

I want a "Share as image" button in the summary view that generates a nice-looking card I can share on WhatsApp — showing who owes what, the split name, and the total.

What happened

A share-as-image feature was built using canvas rendering — generates a settlement card styled in the app's design language, then downloads it as a PNG or triggers the native share sheet on mobile. Perfect for quick WhatsApp sharing.

11 iOS spec

Abhi said

Can you generate a spec.md from this entire app? I want to use it to build an iOS version. Native elements mostly, but keep the visual language — especially the orange accent and the DM Serif headings.

What happened

Claude read the entire app and wrote a comprehensive iOS specification covering all screens, navigation flows, data model, SwiftUI component guidance, and a section on translating the web design tokens into native iOS patterns (system tint, SF Symbols, grouped table views).

Deploy & Ship
12 Firebase setup

Abhi said

I've created a Firebase project — it's called splitsecond-d6d43, region is Asia Southeast. Here's my real config. Can you wire it in?

What happened

The placeholder Firebase config was replaced with the real project credentials. SDK version was confirmed at 12.14.0 to match what the Firebase Console provided. Database security rules were also set up — all reads and writes require authentication.

13 Vercel deploy

Abhi said

I pasted the config and clicked publish. What's next? I have the file — but I don't think the login screens are in the current version yet. I got it running at split-second-weld.vercel.app. Can you add the login bit?

What happened

Claude fetched the live URL to check what was actually deployed, found it was still in demo mode with the placeholder config, and then integrated the full auth flow — login screen, anonymous continue option, Google sign-in, and email sign-up — into the deployed version.

14 OG preview image

Abhi said

I want a nice preview image for when I share the link on WhatsApp — the kind that shows up as a card preview. Can you generate one that matches the app's design?

What happened

An OG preview image was generated using Playwright so it could render actual DM Serif Display from Google Fonts — not a fallback system font. The image shows the SplitSecond wordmark, tagline, and a sample settlement card on the warm parchment background.

og-image.png
15 OG image fix

Abhi said

In that generated image, the "SplitSecond" text doesn't match the serif font we've been using. Fix that. And also remove the little screenshot with the names from it.

What happened

The image was re-rendered via browser canvas to use the actual DM Serif Display font — Pillow (the Python image library) can only use system fonts, so it couldn't match. The screenshot element was removed, leaving just the wordmark, tagline, and feature pills on the warm background.

16 Homescreen icons

Abhi said

Can I make sure that when people add this app to their homescreen on mobile, it has a nice icon? Something that looks like the app — maybe with the two S's from the name.

What happened

Three icon sizes were generated (512px, 192px, 180px) — an orange rounded square with an upright "S" and an italic "S" in DM Serif Display, representing the two words in the name. Meta tags for Apple and Android PWA were added to the HTML. All icons were deployed alongside the app.
